# AgentTest AI 开发协作规范

本文件是本仓库所有 AI Agent 和开发人员必须遵守的仓库级开发协议。

目标是让不同会话、不同 Agent 和不同开发人员始终基于同一产品目标、架构约束和当前进度工作，避免重复开发、遗漏验证、架构漂移和上下文丢失。

---

## 1. 开始任何开发前必须阅读

按以下顺序阅读：

1. `AGENTS.md`
2. `docs/Agent测试平台产品需求文档-PRD.md`
3. `docs/Agent测试平台技术架构与开发规范.md`
4. `docs/开发进度与变更记录.md`
5. 当前任务涉及目录中的 README、ADR、测试和局部 `AGENTS.md`

如果任务涉及特定模块，只需重点阅读相关章节，但不得跳过：

- PRD 的产品定位、权限、数据隔离和验收标准。
- 架构文档的依赖边界、目录规范、禁止事项和架构验收标准。
- 开发进度文档中的当前阶段、已完成工作、进行中任务和已知问题。

在没有完成上述阅读前，不得开始写代码、创建目录、引入依赖或修改数据库。

---

## 2. 文档优先级

发生冲突时按以下优先级处理：

1. 用户当前明确指令。
2. 本文件 `AGENTS.md`。
3. 产品需求文档 PRD。
4. 技术架构与开发规范。
5. 已批准的 ADR。
6. 当前实施计划。
7. 现有代码惯例。

如果 PRD 与技术架构冲突，不得自行选择。先在开发记录中标记冲突，并请求确认或创建 ADR。

## 3. 开发前工作协议

### 3.1 检查当前状态

开发前必须：

- 查看 Git 状态，保留用户已有修改。
- 阅读最新开发进度。
- 搜索是否已有相同功能、接口、组件、迁移或测试。
- 确认任务属于哪个产品模块。
- 确认是否存在前置任务或阻塞问题。
- 确认当前开发是否符合 MVP 范围。

### 3.2 创建任务记录

开始开发前，在 `docs/开发进度与变更记录.md` 的“进行中”部分新增任务：

```markdown
### TASK-YYYYMMDD-序号：任务名称

- 状态：进行中
- 开始日期：YYYY-MM-DD
- 负责人：AI / 姓名
- 关联需求：PRD 章节
- 关联架构：架构文档章节或 ADR
- 目标：一句话描述可验证结果
- 计划修改：文件或模块列表
- 验收方式：测试、命令或人工检查
```

同一时间不得创建名称和目标重复的进行中任务。

### 3.3 复杂任务必须先计划

符合以下任一条件时，必须先编写实施计划：

- 涉及两个及以上业务模块。
- 涉及数据库迁移。
- 涉及认证、权限、凭证或安全。
- 涉及 Workflow、Worker 或插件协议。
- 引入或升级核心开源框架。
- 预计修改超过 5 个主要文件。
- 需求存在多种合理实现。

计划保存在：

```text
docs/superpowers/plans/YYYY-MM-DD-任务名称.md
```

## 4. 开发过程规范

### 4.1 遵守架构边界

必须遵守：

- 前端按 Feature 组织，通过公开出口依赖。
- 后端使用 Domain、Application、Infrastructure、API 分层。
- 模块之间只使用公开接口或事件。
- API 不直接操作 ORM。
- Worker 不直接连接业务数据库。
- 插件只依赖公开 Plugin SDK。
- 所有项目业务数据强制关联并校验 `project_id`。
- 已发布的 Agent、数据集、用例和测试计划使用不可变版本。

详细规则以 `docs/Agent测试平台技术架构与开发规范.md` 为准。

### 4.2 测试驱动和小步提交

- 新功能和缺陷修复先补充失败测试，再实现。
- 每次修改保持单一目标。
- 不顺手重构无关代码。
- 不删除或覆盖用户已有修改。
- 不使用跳过测试、忽略类型或关闭规则掩盖问题。
- 每完成一个独立任务，运行与风险匹配的验证。

### 4.3 依赖管理

- 引入开源框架前核对官方最新稳定版。
- 禁止使用浮动 `latest`、Git `main` 或未固定 Commit。
- 必须提交 Lockfile。
- 记录许可证、安全、兼容性和回滚方案。
- 升级依赖时不得混入无关功能开发。

### 4.4 数据库修改

- 所有 Schema 修改通过 Alembic。
- 迁移文件创建后不得改写已发布历史。
- 使用 Expand → Migrate → Contract。
- 新表和核心查询必须说明项目隔离、外键、唯一约束和索引。
- 数据迁移必须提供验证和回滚/补偿方案。

### 4.5 安全

- 禁止提交密码、Token、Cookie、API Key、测试账号或真实用户数据。
- 日志、Trace 和测试产物必须脱敏。
- 前端隐藏入口不能代替后端权限校验。
- 外部输入、模型输出、Markdown、HTML、URL 和文件均视为不可信。
- 高风险操作必须有确认、权限检查和审计。

## 5. 完成开发前必须验证

不得在没有证据的情况下声称“完成”“修复”或“通过”。

根据改动范围运行：

### 前端

```text
format
lint
typecheck
unit/component tests
critical Playwright E2E
build
```

### 后端

```text
format
ruff/lint
typecheck
unit tests
integration tests
API contract tests
architecture tests
```

### 数据库

```text
空数据库迁移
上一版本数据库升级
约束和索引验证
项目数据隔离测试
```

### Worker / Workflow

```text
Workflow replay
timeout/retry/cancel
idempotency
artifact upload
error classification
```

如果环境原因无法运行某项验证，必须在开发记录中明确写出“未验证”、原因和风险，不能用推测代替。

## 6. 每次开发完成后必须更新记录

完成任务后，必须更新 `docs/开发进度与变更记录.md`：

1. 将任务从“进行中”移动到“已完成”。
2. 记录实际修改的文件和模块。
3. 记录数据库、API、事件和配置变更。
4. 记录执行过的测试及结果。
5. 记录未完成内容、已知问题和技术债。
6. 记录下一步建议。
7. 更新当前里程碑完成度。

完成记录格式：

```markdown
### TASK-YYYYMMDD-序号：任务名称

- 状态：已完成
- 完成日期：YYYY-MM-DD
- 关联需求：PRD 章节
- 实现摘要：
  - ...
- 变更文件：
  - `path/to/file`
- 数据库/API/配置变化：
  - 无 / 具体内容
- 验证记录：
  - `具体命令`：通过
- 遗留问题：
  - 无 / 具体内容
- 下一步：
  - ...
```

代码和记录属于同一个任务。没有更新开发记录，不视为任务完成。

## 7. 开发进度状态定义

只使用以下状态：

| 状态 | 含义 |
|---|---|
| 待开始 | 已明确但尚未开发 |
| 进行中 | 正在开发，同一任务只能有一个主要负责人 |
| 阻塞 | 存在明确外部依赖或决策阻塞 |
| 待验证 | 代码已完成，但关键验证尚未完成 |
| 已完成 | 已满足验收标准并记录验证证据 |
| 已取消 | 明确不再实施，并记录原因 |

禁止把“代码写完但未测试”标记为已完成。

## 8. 文档同步规则

出现以下变化时必须同步文档：

| 变化 | 必须更新 |
|---|---|
| 产品范围、角色、流程或验收变化 | PRD |
| 模块边界、技术栈、目录、数据或安全变化 | 技术架构规范 |
| 重大技术决策 | `docs/adr` |
| API 契约变化 | OpenAPI 和 API 文档 |
| 数据库变化 | Migration 和数据字典 |
| 当前开发状态变化 | 开发进度与变更记录 |
| 部署或运维方式变化 | Runbook |

文档与代码不一致时，任务不得直接关闭。

---

## 9. AI 会话交接规范

每次 AI 开发会话结束前，必须确保下一位 Agent 可以从仓库恢复上下文。

交接至少包含：

- 当前任务状态。
- 已完成内容。
- 修改文件。
- 已运行验证。
- 未完成步骤。
- 阻塞原因。
- 下一步建议。

优先写入 `docs/开发进度与变更记录.md`，不能只保留在聊天记录中。

新会话不得仅依赖用户口述或历史聊天，应以仓库文档和代码为准。

---

## 10. 禁止事项

- 未读 PRD、架构规范和开发进度就开始修改代码。
- 重复实现已有功能。
- 未记录任务就进行大规模开发。
- 擅自改变产品范围或核心架构。
- 为赶进度绕过权限、数据隔离、测试或审计。
- 使用临时条件分支破坏插件扩展体系。
- 把所有业务堆入单个文件、Service 或页面。
- 声称测试通过但没有实际运行。
- 完成开发后不更新进度和变更记录。
- 仅在聊天中说明关键决策而不写入仓库。
