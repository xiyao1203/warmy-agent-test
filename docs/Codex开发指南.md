# Codex 开发指南

本指南说明如何使用 Codex 持续开发 Agent 测试平台，并让不同会话之间保持一致上下文。

Codex 的强制仓库规则位于根目录 `AGENTS.md`。本指南提供实际操作流程和可复制提示词，不替代 `AGENTS.md`。

---

## 1. Codex 为什么可以直接开发本项目

Codex 在开始工作前会读取仓库路径上的 `AGENTS.md`。本项目把稳定、强制的开发规则放在根目录，并把易变化的信息拆分为：

- `docs/当前任务.md`：唯一活动任务和允许修改范围。
- `docs/开发进度与变更记录.md`：长期进度、历史、阻塞和技术债。
- PRD：产品目标和验收标准。
- 技术架构规范：目录、依赖、数据、安全和质量规则。
- 实施计划：复杂任务的逐步执行说明。

这种分层可以让根指令保持稳定，同时避免 Codex 每次从长篇文档中猜测当前任务。

---

## 2. 第一次打开仓库

在仓库根目录运行：

```bash
codex --ask-for-approval never "请列出你加载的项目指令，并用不超过 8 行总结当前任务、允许修改范围、架构约束和验收方式。不要修改文件。"
```

预期结果：

- Codex 能识别根目录 `AGENTS.md`。
- Codex 会读取 `docs/当前任务.md`。
- Codex 能说明当前阶段、任务和禁止修改范围。
- Codex 不会立即创建代码。

若 Codex 没有读取正确指令：

1. 确认从 Git 仓库根目录启动。
2. 运行 `codex status` 检查 Workspace Root。
3. 检查是否存在更高优先级的 `AGENTS.override.md`。
4. 重新启动 Codex 会话。

## 3. 标准开发提示词

### 3.1 开始当前任务

```text
请按 AGENTS.md 执行 docs/当前任务.md 中的当前任务。

开始前：
1. 阅读当前任务、开发进度、相关 PRD、架构和实施计划。
2. 检查 git status、最近提交和已有实现。
3. 用不超过 8 行复述目标、范围、约束、计划和验收方式。

执行中：
- 遵守允许修改范围。
- 先写测试，再实现。
- 不修改无关文件。

完成后：
- 运行规定的验证。
- 更新 docs/当前任务.md 和 docs/开发进度与变更记录.md。
- 汇报变更文件、验证证据、遗留问题和下一步。
```

### 3.2 从待开始任务中选择下一项

```text
请读取 AGENTS.md、docs/当前任务.md 和开发进度。
如果当前无活动任务，从“待开始任务”中选择第一个前置条件已满足的任务。
先解释为什么选择它，并登记当前任务。
复杂任务先创建实施计划，不要直接编码。
```

### 3.3 继续上次未完成的任务

```text
请恢复当前任务。
重新读取 docs/当前任务.md，检查 git status、当前 diff 和最近 3 条提交。
先说明已经完成什么、还剩什么、是否存在冲突，再继续。
不要依赖聊天记忆代替仓库状态。
```

### 3.4 代码评审

```text
请按 PRD、技术架构规范和 AGENTS.md 评审当前变更。
重点检查：
- 产品验收是否覆盖
- 权限和 project_id 隔离
- 模块依赖方向
- API/数据库兼容性
- 幂等、超时、重试和错误分类
- 测试和文档记录

先报告问题，不要直接修改，除非我明确要求修复。
```

### 3.5 会话交接

```text
请停止新增功能，为下一次 Codex 会话准备交接。
更新 docs/当前任务.md 和开发进度记录，写清：
- 已完成内容
- 修改文件
- 验证结果
- 未完成步骤
- 阻塞条件
- 下一条建议操作
不要只在回复中说明。
```

## 4. 推荐开发节奏

每个任务遵循：

```text
读取上下文
→ 登记任务
→ 编写实施计划（复杂任务）
→ 写失败测试
→ 最小实现
→ 运行验证
→ 自检架构与安全
→ 更新文档
→ 提交
→ 交接
```

建议一个 Codex 会话只完成一个主任务。大型功能拆成多个可以独立验证的任务，不要让一个会话同时搭建前端、后端、数据库和 Worker 的全部功能。

---

## 5. 任务文件规则

`docs/当前任务.md` 必须保持短小，只包含当前一个任务：

- 当前阶段和状态。
- 任务目标。
- 允许修改文件或目录。
- 禁止修改范围。
- 验收条件。
- 阻塞和下一步。

完成后：

1. 将完整结果写入开发进度的“已完成任务”。
2. 将当前任务设置为“当前无活动任务”，或登记用户明确要求的下一任务。
3. 不把历史任务持续堆积到当前任务文件。

---

## 6. 复杂任务与实施计划

复杂任务的实施计划放在：

```text
docs/superpowers/plans/YYYY-MM-DD-任务名称.md
```

计划应包含：

- 精确文件路径。
- 每一步的测试和实现。
- 执行命令和预期结果。
- 数据库迁移。
- API 和事件变化。
- 文档同步。
- 独立提交点。

Codex 执行计划时，每完成一组任务就验证，不要积累到最后一次性测试。

---

## 7. 子目录 AGENTS.md

项目开始产生代码后，在专业目录建立局部指令：

```text
AGENTS.md                         # 全仓库规则
apps/web/AGENTS.md                # 前端命令和 UI 规则
apps/control-api/AGENTS.md        # 后端分层和迁移规则
workers/AGENTS.md                 # Worker 和 Temporal 规则
plugins/AGENTS.md                 # Plugin SDK 规则
```

Codex 会把从仓库根到当前工作目录的指令合并，越接近当前目录的规则优先级越高。

局部文件只写差异：

- 该目录构建和测试命令。
- 目录专属架构边界。
- 专属验收和禁止事项。

不要复制根 `AGENTS.md`，避免规则重复和冲突。

---

## 8. Codex 工作树和并行开发

多个独立任务需要并行时，优先使用独立 Git Worktree 或 Codex Thread：

- 每个 Worktree 对应一个明确任务和分支。
- 不允许两个任务同时修改同一迁移、公共契约或进度台账。
- 合并前分别运行验证。
- 主任务负责整合开发进度记录。

权限、安全、数据库 Schema 和公共契约等高耦合任务不建议并行。

---

## 9. 完成定义

Codex 只有在以下条件全部满足后才能声明完成：

- 需求验收条件满足。
- 架构边界未被破坏。
- 代码、迁移、配置和文档一致。
- 规定测试实际运行并通过。
- 未执行的验证已明确标注风险。
- 当前任务和开发进度已更新。
- Git Diff 不包含无关改动和敏感信息。

“代码已生成”不等于完成。

---

## 10. 官方依据

本仓库使用根目录和子目录 `AGENTS.md` 的方式，遵循 OpenAI Codex 官方的项目指令发现和覆盖机制：

- Codex 在工作前读取 `AGENTS.md`。
- 从仓库根目录向当前工作目录逐层读取。
- 更接近当前目录的指令优先。
- `AGENTS.override.md` 可以覆盖同目录的普通指令。
- 项目指令默认存在合并大小限制，应保持根指令精简并使用局部文件。

官方文档：

- <https://developers.openai.com/codex/guides/agents-md>
