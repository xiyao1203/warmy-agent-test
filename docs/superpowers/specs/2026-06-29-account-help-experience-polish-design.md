# 账户与帮助体验优化设计说明

## 1. 目标与设计方向

本任务采用已确认的 **A 方案：统一账户中心**，在不改变“方案 1：极简控制台”整体视觉语言的前提下，统一优化账户、偏好、通知、帮助与反馈体验。

核心目标：

- 将分散的个人资料、偏好、通知和安全设置收拢为结构稳定的账户中心。
- 让顶部帮助、通知和用户菜单具备完整、可理解、可键盘操作的交互。
- 将帮助中心从静态展示页升级为可搜索、可导航、可反馈的支持空间。
- 接通已有后端资料、密码、用户设置和反馈 API，不保留“看起来可点但没有行为”的控件。
- 延续高密度研发工具风格，并保证窄屏、加载、错误、空状态和危险操作可用。

## 2. 现状审计

审计证据保存在：

- `docs/ui-audits/2026-06-29-account-experience/help-center-current.png`
- `docs/ui-audits/2026-06-29-account-experience/feedback-current.png`

当前主要问题：

1. 顶部通知按钮没有实际行为；帮助和用户菜单仅处理鼠标点击，缺少完整菜单键盘与焦点管理。
2. 帮助菜单把内部页面以新标签打开，并包含占位 GitHub 地址，信息架构和可信度不足。
3. 帮助中心的“搜索”只是视觉占位，多个模块链接指向不存在的锚点；窄屏存在内容裁切和横向溢出。
4. 个人资料中的编辑头像、编辑资料、修改密码、两步验证和删除账号入口大多没有实际交互。
5. 设置页仅保存在组件本地状态，保存、重置和后端用户设置 API 没有形成闭环。
6. 个人资料与设置是两个孤立页面，安全、通知、偏好信息重复分散，缺少当前位置和稳定二级导航。
7. 反馈页只切换本地成功状态，未接通已有反馈 API，也缺少提交中、失败重试和防重复提交状态。
8. 当前后端启动被 `identity/api/router.py` 中段重复 `from __future__` 导入阻断；`bootstrap/app.py` 也存在明显重复定义，必须先做范围受控的恢复检查。

## 3. 信息架构

### 3.1 统一账户中心

新增统一入口 `/account`，通过 `section` 查询参数切换以下区域：

- `profile`：头像、显示名称、邮箱、用户 ID、角色和账号状态。
- `preferences`：主题、语言、减少动态效果等界面偏好。
- `notifications`：邮件、浏览器推送、测试完成和告警通知偏好。
- `security`：修改密码、两步验证状态、活跃会话和危险操作。

兼容策略：

- `/profile` 重定向至 `/account?section=profile`。
- `/settings` 重定向至 `/account?section=preferences`。
- 顶部用户菜单直接链接账户中心对应区域。

桌面端使用左侧二级导航和右侧内容区；窄屏使用可横向滚动的紧凑分段导航，不允许页面主体横向溢出。

### 3.2 帮助中心

保留 `/docs`、教程、测试指南、快捷键和反馈路由，统一使用帮助中心布局：

- 顶部返回应用、帮助中心标题和真实搜索框。
- 桌面端左侧目录、主内容和可选右侧页内目录。
- 窄屏收拢目录，内容单列展示。
- 搜索至少支持标题、摘要和关键词过滤；无结果时提供清空条件与反馈入口。
- 内部链接在当前标签打开；外部链接只在配置为真实地址时展示并明确标识。

### 3.3 顶部入口

- 帮助按钮：打开紧凑 Popover，提供搜索、常用入口、快捷键和反馈，不复制完整帮助中心。
- 通知按钮：打开通知 Popover，包含未读状态、空状态和“通知偏好”入口；本期无通知流 API 时不得伪造消息。
- 用户菜单：展示身份摘要、账户中心入口、偏好入口和退出登录；菜单焦点关闭后回到触发按钮。

## 4. 关键交互

### 4.1 资料编辑

- 默认以只读摘要展示；点击“编辑资料”进入局部编辑态。
- 显示名称支持校验、保存、取消、提交中和服务端错误。
- 邮箱若后端不允许修改，则明确标注为登录邮箱，不渲染伪编辑入口。
- 头像上传只有在后端契约支持时开放；否则移除无行为的相机按钮，保留首字母头像。
- 复制用户 ID 使用 Toast/可访问状态反馈，不依赖短暂文字替换作为唯一反馈。

### 4.2 偏好与通知设置

- 首次进入读取后端用户设置。
- 修改后显示“有未保存更改”，保存成功后更新查询缓存并反馈。
- 重置仅恢复最近一次服务端值；离开有未保存更改的区域时提示确认。
- 主题选择立即预览，但只有保存成功后作为持久偏好；不支持的深色主题不得做伪开关。
- 浏览器推送必须在用户主动开启时请求浏览器权限，并处理拒绝、不可用和已阻止状态。

### 4.3 安全操作

- 修改密码使用 Dialog，字段包含当前密码、新密码、确认密码和显示切换。
- 客户端校验和后端错误就近展示，成功后说明会话影响。
- 两步验证若暂无后端能力，展示“暂未开放”说明，不提供可点击假入口。
- 删除账号只有在后端契约和权限策略明确支持时展示；危险确认必须说明不可恢复影响并要求明确确认。

### 4.4 帮助与反馈

- 搜索输入支持清空、键盘提交和结果数量反馈。
- FAQ 使用可访问 Accordion，展开状态可键盘操作。
- 反馈表单接入后端 API，支持必填校验、最小长度、提交中、防重复提交、失败重试和成功回执。
- 成功后允许返回帮助中心或继续提交另一条反馈。

## 5. 视觉规范

- 继续使用现有 `--background`、`--surface`、`--surface-subtle`、`--border`、`--text-muted` 和 `--accent` Token。
- 页面标题控制在 24 像素左右，正文以 13–14 像素为主；不增加营销式大标题或装饰性渐变。
- 内容最大宽度与现有工作台对齐，账户中心桌面宽度约 960–1040 像素。
- 卡片只用于稳定分组；设置项优先使用分隔行，减少重复大卡片造成的纵向浪费。
- Popover、Dialog 和 Drawer 使用克制阴影；状态颜色只表达真实状态。
- 图标统一使用现有 Lucide 图标库，不使用文字符号代替。

## 6. 前后端边界

前端：

- 新建账户 Feature，通过公开出口组合账户中心页面。
- 平台布局仅管理弹层呈现，不承载资料、设置或反馈业务规则。
- 页面通过生成的 API Client 读取和提交数据；Mutation 携带 Cookie 与 CSRF Token。
- 使用现有公共 `Button`、`Input`、`Dialog`、`Popover`、`Tooltip`、`Toast` 等组件，补充缺失能力时保持无业务依赖。

后端：

- 复用身份模块的资料更新和密码修改能力。
- 复用用户设置模块的读取与更新能力。
- 复用反馈模块的创建能力。
- 先修复当前重复代码导致的启动错误，并通过回归测试确认没有改变既有 API 契约。
- 本期不新增通知消息领域模型；通知 Popover 只展示真实可用状态和偏好入口。

## 7. 状态、无障碍与响应式

- 所有异步区域具有结构化 Skeleton、局部错误、重试和空状态。
- 所有图标按钮有可访问名称和 Tooltip；菜单、Dialog、Popover 管理焦点和 Escape 关闭。
- Switch 具备 `role="switch"`、`aria-checked` 和可见文字，不只依赖颜色。
- 表单 Label、说明和错误信息使用正确关联。
- 1280 像素以上展示完整账户二级导航；768–1279 像素压缩间距；小于 768 像素使用单列并保证无横向溢出。
- 尊重 `prefers-reduced-motion`，动效保持 150–200ms 的克制反馈。

## 8. 测试与验收

先写失败测试，再实现。至少覆盖：

- 后端可启动，身份资料、密码、用户设置和反馈 API 回归通过。
- 账户中心二级导航、兼容重定向和窄屏布局。
- 资料编辑成功、校验失败、服务端失败和取消编辑。
- 设置加载、脏状态、保存、重置和浏览器推送权限分支。
- 密码 Dialog 的校验、提交状态、成功与失败。
- 帮助搜索、无结果、FAQ 键盘操作和反馈完整提交状态。
- 顶部帮助、通知、用户菜单的键盘导航、外部点击、Escape 和焦点恢复。
- 个人资料、账户中心、帮助中心和反馈页的关键 Playwright E2E。

## 9. 非目标

- 不建立新的通知消息存储、已读状态或实时推送系统。
- 不新增两步验证后端协议、头像对象存储或账号删除策略。
- 不重做全局侧栏、Agent 对话流式页面或其他业务模块。
- 不改变现有方案 1 的品牌色、字体体系和全局信息密度。

## 10. 通过条件

- 用户可从顶部用户菜单进入统一账户中心，并在一个稳定结构内完成资料、偏好、通知和安全操作。
- 顶部帮助、通知、用户菜单不存在无响应控件，且键盘和焦点行为完整。
- 帮助中心搜索、导航、FAQ 和反馈均可实际使用。
- 现有后端能够正常启动，相关 API 契约与权限检查保持有效。
- 全部相关验证通过，并完成桌面与窄屏视觉对比记录。
