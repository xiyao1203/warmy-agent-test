# Agent 对话 Codex 式全链路体验优化设计

## 1. 背景与目标

现有 Test Agent 已支持消息流、思考增量、工具事件、确认卡片和会话恢复，但过程事件不是会话恢复模型的一部分；停止按钮只中止浏览器请求；新会话的 SSE 与发送请求存在首包竞态；工具过程在完整回答后集中显示。结果是 UI 看似实时，事实状态却可能在刷新、停止或弱网场景下分叉。

本任务将对话改造成一个可恢复、可取消、按真实顺序呈现的执行界面：正文采用 ChatGPT 式低噪声阅读体验，思考摘要、工具调用和结果采用 Codex 式内联时间线。

## 2. 范围

包含：

- 超级 Test Agent 会话的生成生命周期、事件恢复、SSE 游标和取消。
- 消息、思考摘要、工具调用、确认、产物、错误和取消状态的统一时间线。
- 新会话握手、断线重连、刷新恢复和多标签页下的服务端一致性。
- 对话页的视觉层级、状态反馈、键盘与读屏语义。
- 与上述行为直接相关的迁移、API 契约、测试和文档。

不包含：

- 改造独立“被测 Agent 对话”协议。
- 展示模型私有思维链或供应商原始 reasoning token。
- 用 WebSocket 替换现有 SSE。
- 重做平台全局导航或其他业务页面。

## 3. 体验设计

### 3.1 视觉方向

- 采用已确认的 A 方案：ChatGPT 式正文 + Codex 式内联过程时间线。
- 主阅读列保持约 48rem，用户消息使用克制的浅色圆角块，助手正文不使用厚重卡片。
- 过程条目使用细线、图标、短标题和状态文字组织；默认展示行动摘要，参数、输出与错误详情可展开。
- 移除按消息数增长的装饰性进度条；保留必要的会话、上下文和产物入口。
- 颜色只表达运行中、成功、等待确认、失败和取消，不用大面积装饰色。

### 3.2 时间线顺序

一次生成按服务端事件顺序展示：

1. 用户消息。
2. `generation.started`。
3. 可选的安全思考摘要 `agent.reasoning`。
4. `tool.started` / `tool.confirmation_required`。
5. `tool.completed`、`tool.failed` 或 `tool.cancelled`。
6. 助手增量与最终消息。
7. `generation.completed`、`generation.failed` 或 `generation.cancelled`。

流式文字可以持续更新同一条助手草稿，但完成前不得写成正式消息。刷新后只恢复持久化的安全摘要和工具事件，不回放原始文本 delta。

### 3.3 明确状态

输入区和时间线必须区分：连接中、正在思考、正在输出、正在调用工具、等待确认、取消中、已取消、正在恢复连接、失败。完成状态只由服务端终止事件驱动，禁止使用无增量超时猜测完成。

## 4. 领域与持久化设计

新增 `ChatGeneration` 作为每次回复生成的领域实体，至少包含：

- `generation_id`、`project_id`、`session_id`。
- `workflow_id`：Temporal Workflow 启动后写入。
- `status`：`pending | running | cancelling | completed | cancelled | failed`。
- `partial_content` 与最终消息关联信息。
- `started_at`、`updated_at`、`completed_at`。

所有业务记录继续强制包含并校验 `project_id`。Schema 通过新的 Alembic 迁移增加，不改写历史迁移。取消和完成使用条件更新，保证终止状态幂等。

现有 `AgentEvent` 保持有序事实日志角色。事件增加 `generation_id`，并保留服务端序号与时间。历史读取只返回可审计语义事件；`message.delta` 和 `agent.reasoning_delta` 不作为长期 UI 条目回放，以控制数据量。

## 5. API 与事件协议

### 5.1 会话详情

`GET /sessions/{session_id}` 增加：

- `timeline`：按服务端时间与稳定序号排序的持久消息和语义事件。
- `event_cursor`：当前最大事件序号。
- `active_generation`：存在时返回生成 ID、状态和可取消能力。

前端不得自行把消息数组和临时事件重新猜序；统一消费时间线 DTO。

### 5.2 SSE 握手与补发

`GET /sessions/{session_id}/events?after=<cursor>` 明确使用游标：

- 新会话创建响应返回初始 `event_cursor`。
- EventSource 建连后服务端先发 `stream.ready`，携带实际游标。
- 前端收到 `stream.ready` 后才允许发送消息。
- 重连使用最后确认的事件序号，服务端补发其后的事件。
- 心跳不推进业务游标；断线不改变生成状态。

### 5.3 发送与取消

发送请求携带客户端生成的 `generation_id`，服务端按项目、会话、生成 ID 幂等创建生成记录。重复请求返回同一生成状态，不启动第二个 Workflow。

新增 `POST /sessions/{session_id}/generations/{generation_id}/cancel`：

- 校验项目、会话、写权限和 CSRF。
- 把状态原子更新为 `cancelling`，调用 Model Invoker 的 Temporal 取消能力。
- Worker 确认取消后写入部分内容，发出 `generation.cancelled`。
- 已终止生成重复取消返回当前状态，不报假失败。

浏览器 Abort 仅负责释放当前 HTTP 等待，不再代表业务取消。

## 6. 后端执行链路

Control API 在发送请求中创建生成记录并发出 `generation.started`。Model Invoker 启动 Temporal Workflow 后，将真实 `workflow_id` 回写生成记录。模型回调继续写入 delta 事件，同时节流更新 `partial_content`，避免每个 token 都更新生成表。

规划与工具执行不能再表现为“回答完成后的附加动作”。服务端先产生可审计行动计划摘要，再委派工具；工具完成后生成面向用户的最终回答。若当前模型能力需要两阶段调用，第一阶段只用于计划和安全摘要，不作为最终回答；第二阶段基于工具结果生成最终正文。

工具执行事件统一映射为开始、等待确认、完成、失败和取消。高风险写操作继续经过现有确认领域模型，不因 UI 改造绕过权限或审计。

## 7. 前端状态模型

前端 reducer 拆分为：

- 会话快照：服务端返回的 timeline、cursor、active generation。
- 连接状态：`connecting | ready | reconnecting | offline`。
- 生成状态：完全镜像服务端生成状态。
- 临时流：当前助手草稿和当前安全思考摘要。

收到重复事件时以事件序号去重；收到快照时用服务端 cursor 替换本地基线，再合并更大的已接收事件。任何会话切换都关闭旧连接并隔离旧回调。

“停止生成”先进入 `cancelling`，调用取消 API，收到终止事件后进入 `cancelled`。部分回答带“已停止”标识，不提供重新生成之外的虚假成功操作。

## 8. 无障碍与响应式

- 消息日志使用稳定的 `role=log`，只让新增状态通过专用 live region 宣告，避免整个历史重复朗读。
- 思考和工具详情使用原生 button/region 语义，具备 `aria-expanded` 和可见焦点。
- 状态不能只依赖颜色；图标旁始终提供文字。
- 停止、重试、确认和展开控件满足键盘操作与合理点击区域。
- 右侧上下文面板在窄屏改为可打开的抽屉，主时间线保持完整。

## 9. 错误与边界场景

- SSE 在生成中断线：保留草稿，显示“正在恢复连接”，按 cursor 补发后继续。
- 页面刷新：从会话详情恢复时间线、待确认和活动生成，再从 cursor 订阅。
- 取消与完成竞态：第一个成功写入的终止状态为准；接口返回真实最终状态。
- 工具失败：保留失败条目和安全错误摘要，最终回答解释未完成范围。
- 缺少默认模型或运行时：生成标记失败并返回稳定问题详情，不留下永久运行中状态。
- 多标签页：相同 generation ID 幂等；不同 generation ID 是否并发由服务端会话策略拒绝或排队，首版同一会话只允许一个活动生成。

## 10. 测试与验收

后端：

- 生成实体状态机、项目隔离、幂等和终止竞态单元测试。
- 会话详情恢复语义事件、待确认与活动生成的契约测试。
- SSE ready、游标补发和重复事件测试。
- Temporal Workflow ID 回写、真实取消、部分内容保存与错误收敛测试。
- 空库迁移、上一版本升级、约束和索引验证。

前端：

- reducer 的快照合并、事件去重、会话隔离和连接状态测试。
- 新会话必须等待 `stream.ready` 才发送。
- 刷新恢复工具/确认时间线、停止生成和断线重连组件测试。
- 键盘、ARIA 和响应式关键状态检查。

验收场景：

1. 新建会话并看到首个 delta，无首包丢失。
2. 工具过程按真实顺序内联呈现。
3. 等待确认时刷新，确认卡仍存在且可决策。
4. 输出中停止，Temporal 被取消，费用和状态不继续增长，刷新后仍显示已取消的部分回答。
5. SSE 断线重连后不重复、不缺失语义事件。
6. 完整质量门禁通过；真实模型 E2E 若受外部凭证限制，明确记录未验证风险。

## 11. 迁移与回滚

采用 Expand → Migrate → Contract：先新增 generation 表和可空事件关联字段，再让新代码双向兼容旧事件。回滚应用版本时新增表和字段不会影响旧读取路径；数据库 downgrade 仅用于开发与测试环境。发布后不改写该迁移历史。
