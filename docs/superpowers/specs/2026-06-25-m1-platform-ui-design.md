# M1 平台界面设计说明

## 1. 目标

为 M1 的登录、用户管理、项目切换和项目概览建立统一、可扩展的界面基础，并确保后续 Agent 流式对话、测试计划配置、运行进度和 Trace 工作台可以自然接入同一应用框架。

本设计采用已确认的“方案 1：极简控制台”视觉方向：

- Linear / Vercel 式专业研发工具语言。
- 桌面端优先，高信息密度、低视觉噪声。
- 浅色主题优先，中性色为主，品牌色和状态色克制使用。
- 主要依靠排版、间距、对齐和分隔线建立层级。
- 管理页面使用高密度表格；工作页面允许三栏布局。

## 2. 本期范围

### 2.1 本期实现

- 邮箱和密码登录。
- 未登录访问保护与登录后返回原地址。
- 全局平台壳：顶部栏、项目导航、系统管理导航和用户菜单。
- 授权项目切换。
- 项目概览与成员摘要。
- 超级管理员用户列表、筛选、详情、创建、编辑、重置密码、启用、禁用和删除入口。
- 普通用户隐藏系统管理入口，并阻止进入用户管理页面。
- 页面加载、空数据、无搜索结果、无权限和服务错误状态。

### 2.2 本期不实现

- Agent 对话业务逻辑、流式协议和消息持久化。
- 测试计划生成、运行进度、Trace 和结果分析。
- M2 之后的 Agent、数据集、用例和运行模块。

本期仍需预留稳定的工作区结构，使后续 Agent 对话页无需重做全局导航和视觉体系。

## 3. 全局应用框架

### 3.1 顶部栏

顶部栏高度保持紧凑，包含：

1. 产品名称。
2. 当前项目切换器。
3. 全局搜索入口，M1 未接入搜索能力时不渲染。
4. 运行状态入口，M1 未接入运行能力时不渲染。
5. 当前用户菜单。

项目切换器只展示当前用户有权访问的项目。切换项目时保持当前功能位置；若目标项目不存在对应资源，则回到目标项目概览。

### 3.2 左侧导航

左侧导航分为两个明确区域：

- 项目工作区：概览、测试 Agent、Agent 与版本、测试集、测试计划、运行记录等。
- 系统管理：用户管理和全局审计，仅超级管理员可见。

M1 尚未开放的后续模块可以显示为不可用导航项或暂不渲染，但不得伪造可操作页面。

桌面宽度下显示图标和文字；1024–1279 像素可折叠为图标栏。左侧导航不承担页面级筛选或对象详情。

### 3.3 主工作区

主工作区根据页面类型使用两类布局：

- 管理页：标题与操作区、筛选区、数据表格、可选右侧详情抽屉。
- 工作台页：左侧导航、中间主任务区、右侧上下文面板。

两类布局共享顶部栏、导航、Token、组件和状态反馈，不建立两套视觉系统。

## 4. Agent 流式对话兼容设计

Agent 对话页未来使用全局框架中的工作台布局：

- 中间区域是主要流式消息与任务过程，宽度优先。
- 右侧上下文面板展示 Agent 版本、数据集、执行模式、评分器、预算和阈值。
- 对话输入固定在中间区域底部，但不遮挡消息和持久运行状态。
- 工具调用、计划草稿、审批、运行进度和结果摘要使用专用结构化组件，不用纯文本模拟。
- 流式输出只更新当前消息或过程组件，避免整页跳动。
- 用户滚动离开底部时停止自动跟随，并提供“返回最新”入口。
- 长任务状态保留在页面内，不以 Toast 作为唯一反馈。

右侧面板可以折叠，但涉及成本、安全和结果判定的关键配置必须在运行前可见。该布局约束将影响本期 `AppShell` 的 DOM 和尺寸设计，但本期不提前实现对话业务组件。

## 5. 页面设计

### 5.1 登录页

- 使用单一紧凑登录区域，不放营销插画或大面积装饰。
- 字段为邮箱和密码，提供清晰 Label、错误说明和密码显示切换。
- 客户端先校验必填和邮箱格式。
- 服务端认证失败统一显示通用错误，不泄露账号是否存在。
- 提交期间按钮显示进行中状态并防止重复提交。
- 登录成功后跳转至原请求地址；没有返回地址时进入第一个可访问项目。

### 5.2 用户管理页

页面采用已选方案 1 的高密度结构：

- 页面标题、简短说明和“创建用户”主操作。
- 紧凑指标摘要可展示总用户、活跃、禁用和需改密数量；没有可靠后端数据时不伪造指标。
- 筛选区支持关键词、角色和状态。
- 表格至少展示姓名、邮箱、系统角色、状态、项目关系和操作。
- 行点击打开右侧详情抽屉。
- 低频操作放入行内菜单；危险操作要求确认。

右侧详情抽屉展示：

- 姓名、邮箱、角色和状态。
- 项目成员关系。
- 修改基本信息入口。
- 重置密码、启用、禁用和删除等受权限保护的操作。

当前超级管理员不得对自己执行导致失去管理能力的操作。最后一个可用超级管理员的禁用、降权和删除操作必须明确不可用，并说明原因。

### 5.3 创建与编辑用户

- 使用 Dialog，不跳转到独立页面。
- 字段错误就近展示，提交失败保留用户输入。
- 创建成功后刷新列表并打开新用户详情。
- 密码通过安全方式展示一次或按后端契约处理，不写入日志和持久前端状态。
- 编辑操作只开放后端允许修改的字段。

### 5.4 密码重置与危险操作

- 密码重置必须展示影响：已有 Session 将失效，用户下次登录需按策略修改密码。
- 禁用用户必须展示受影响用户及 Session 失效结果。
- 删除只在后端允许物理删除时开放；有历史业务数据的用户使用禁用语义。
- Dialog 的确认按钮使用明确动作文本，不使用模糊的“确定”。

### 5.5 项目切换器

- 放置于顶部栏产品名称之后。
- 支持项目名称搜索。
- 只展示当前用户有权访问的项目，并区分活跃和归档状态。
- 当前项目有清晰选中标识。
- 超级管理员可看到创建项目入口；普通用户不可见。
- 键盘可打开、搜索、移动选择和确认。

### 5.6 项目概览

M1 项目概览只展示已有真实数据：

- 项目名称、状态、描述和基础元数据。
- 当前用户的项目权限。
- 成员数量与成员列表摘要。
- 超级管理员的项目设置和成员管理入口。
- 尚未开始的测试业务使用明确空状态，不伪造运行指标。

## 6. 视觉与组件规范

### 6.1 视觉规则

- 页面背景使用 `--background`，主要内容使用 `--surface`。
- 分组优先使用留白和分隔线，必要时使用 `--surface-subtle`。
- 品牌强调使用 `--accent`；成功、警告和危险只表达真实状态。
- 阴影仅用于菜单、Popover、Dialog 和 Drawer。
- 正文以 13–14 像素为主，页面标题不超过 24 像素。
- 表格行高保持紧凑但满足可点击目标和可读性。
- 不使用渐变、玻璃拟态、大插画、巨型卡片或装饰性动画。

### 6.2 公共组件

本期公共组件至少包括：

- `Button`
- `Input`
- `Badge`
- `Dialog`
- `Drawer`
- `DropdownMenu`
- `Table`
- `EmptyState`
- `InlineAlert`
- `Toast`
- `Skeleton`

页面只能组合公共组件和 Feature 公开出口，不复制组件内部实现。

## 7. 前端模块边界

- `features/auth`：登录表单、会话读取、登录退出操作和访问保护。
- `features/users`：用户列表、筛选、详情和管理操作。
- `features/projects`：项目切换、概览和成员摘要。
- `lib/permissions`：纯权限判断，不发起网络请求。
- `components/layout`：平台框架，不包含身份或项目业务规则。
- `components/ui`：无业务依赖的公共组件。
- App Router 页面只负责路由参数、数据边界和 Feature 组合。

所有服务端数据通过生成的 API Client 访问。Mutation 统一附加 CSRF Token，并携带 Cookie 凭证。

## 8. 数据流与权限

1. 平台布局读取当前 Session。
2. 未认证时跳转 `/login?returnTo=<encoded path>`。
3. 已认证时读取当前用户与授权项目列表。
4. 当前项目来自 URL；URL 项目不在授权列表时显示 404 或权限状态，不自动泄露项目存在性。
5. 项目切换更新 URL 并重新加载项目上下文。
6. 系统管理导航和用户管理数据请求同时校验超级管理员权限。
7. 前端隐藏入口只用于体验，最终权限由后端响应决定。

## 9. 状态与错误处理

每个异步 Feature 明确定义：

- 首次加载：使用结构匹配的 Skeleton。
- 局部刷新：保留现有内容并显示轻量进度。
- 空数据：解释原因并给出有权限的下一步操作。
- 无搜索结果：保留筛选条件并提供清除筛选。
- 无权限：说明当前账号无权执行操作，不展示敏感对象信息。
- 网络或服务错误：就近展示错误和重试入口。
- 冲突：提示数据已变化并要求刷新，不静默覆盖。

Toast 只用于轻量成功反馈；表单错误、权限错误和危险操作结果必须在相关区域持续可见。

## 10. 无障碍与响应式

- 目标满足 WCAG 2.2 AA。
- 所有输入具有 Label、错误描述和正确 ARIA 关联。
- 图标按钮具有可访问名称和 Tooltip。
- Dialog、Drawer 和菜单管理焦点，关闭后恢复到触发元素。
- 状态不只依赖颜色，同时提供文字或图标。
- 支持键盘完成项目切换、列表筛选和用户管理主要操作。
- 1280 像素以上展示完整工作区；1024–1279 像素折叠侧栏或右侧面板；小于 1024 像素保证登录、查看和审批基础能力。

## 11. 测试与验收

组件测试按 TDD 实现，至少覆盖：

- 登录字段校验、提交状态和通用认证错误。
- 未认证跳转及 `returnTo` 保留。
- 用户列表加载、空、错误、无权限和有数据状态。
- 创建用户 Dialog 和密码重置确认。
- 当前超级管理员保护和最后管理员保护的前端状态。
- 项目切换器只显示授权项目。
- 普通用户不显示系统管理导航。
- Drawer、Dialog 和项目切换器的键盘与焦点行为。

实现后运行：

```text
pnpm --filter @warmy/web lint
pnpm --filter @warmy/web typecheck
pnpm --filter @warmy/web test
pnpm --filter @warmy/web build
```

## 12. 验收结论

本设计通过以下条件即视为可以进入实施：

- 方案 1 的视觉语言在登录、管理和项目页面保持一致。
- 全局框架明确支持未来 Agent 三栏流式工作区。
- 本期不提前实现 M2 业务，也不伪造测试运行数据。
- 权限、异步状态、危险操作和无障碍行为均有明确规则。
- 组件、Feature、路由和 API Client 边界清晰。
