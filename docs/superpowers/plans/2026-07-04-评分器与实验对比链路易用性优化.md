# 评分器与实验对比链路易用性优化

## 现状待确认

- 评分器管理页应能创建规则、参考答案和模型评分器，并供测试计划版本选择。
- 测试计划版本应把选中的评分器写入配置，执行时由运行链路带入评分任务。
- 实验对比应能基于两次运行创建对比，并查看统计、退化项和逐用例差异。
- 发布门禁和人工审核应能消费运行、评分或实验结果，但本任务不改变门禁规则模型。

## 链路确认

- 评分器 CRUD 接口已存在，但执行链路读取的是 `scorer_versions.id`；原前端测试计划拿到的是 `scorers.id`，会导致执行时无法加载评分器配置。
- 本任务需要让评分器保存时自动生成已发布版本，并让测试计划选择已发布评分器版本。
- 实验对比 API 已校验两次运行属于同一项目和同一测试计划版本，并基于 `run_cases` 生成逐用例对比和统计摘要。

## 问题

- 评分器页面如果只是资产列表，用户不容易理解“创建后在哪里使用、运行后在哪里看结果”。
- 测试计划里的评分器选择需要说明来源和用途。
- 实验对比列表需要把“先运行两次，再创建对比，再查看退化”的路径说清楚。
- 实验对比页面需要提供回到运行中心、测试计划和发布门禁的自然入口。
- 操作文案应避免“查看”等泛词，改成“配置评分 / 创建对比 / 查看对比结果”等明确动作。

## 计划

1. 审计评分器 API、测试计划配置、运行任务和实验对比 API 的真实链路。
2. 修复评分器版本链路：创建/更新评分器时发布可执行版本，迁移回填历史评分器版本。
3. 优化评分器列表：增加闭环说明、空状态引导和下一步入口。
4. 优化测试计划版本向导：在评分器选择处说明“来自评分器页面，运行后进入结果和实验对比”。
5. 优化实验对比列表与详情：增加运行中心、测试计划、发布门禁的入口和更易懂的空状态。
6. 补充前端和迁移测试覆盖关键链路。
7. 运行相关测试、格式、ESLint、TypeScript 和构建。

## 验收

- 用户能从评分器页理解：创建评分器、在测试计划中使用、运行后查看评分结果。
- 用户能从实验对比页理解：选择两次运行创建对比、查看退化、再进入发布门禁。
- 前端关键操作不再依赖泛泛的“查看”文案。
- 不破坏已有评分器、测试计划和实验对比 API 契约。
