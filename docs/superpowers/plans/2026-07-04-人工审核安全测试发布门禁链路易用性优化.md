# 人工审核、安全测试与发布门禁链路易用性优化

## 现状待确认

- 人工审核应承接低置信度、评分冲突、高风险或运行失败结果，并能回到运行详情查看证据。
- 安全测试应围绕测试计划、运行结果和安全发现闭环，不能只是独立列表。
- 发布门禁应能消费运行结果、评分结果、安全发现、人工审核和实验对比结论，并能被测试计划版本选择。

## 问题

- 三个页面如果只展示列表，用户不知道数据从哪里来、下一步去哪。
- 测试计划配置里对发布门禁、人工审核和安全扫描的说明不足。
- 发布门禁、人工审核、安全测试之间需要显性互相跳转，形成“运行结果 → 风险处理 → 发布判断”的路径。
- 操作文案应减少泛词，优先使用“处理审核 / 查看证据 / 配置门禁 / 查看安全发现”等明确动作。

## 计划

1. 审计人工审核、安全测试、发布门禁 API 与运行、测试计划、实验对比的真实链路。
2. 优化人工审核队列：增加来源说明、证据入口和空状态引导。
3. 优化安全测试页：增加从测试计划/运行中心进入安全发现、再进入门禁的路径。
4. 优化发布门禁页：增加“运行结果、实验对比、安全发现、人工审核、发布判断”的路径说明。
5. 优化测试计划版本向导：说明门禁、安全和人工审核如何参与发布。
6. 补充前端测试覆盖关键文案、入口和空状态。
7. 运行相关测试、格式、ESLint、TypeScript 和构建。

## 验收

- 用户能从人工审核页理解任务来源，并回到运行证据或发布门禁。
- 用户能从安全测试页理解如何通过测试计划和运行产出安全发现，并进入门禁。
- 用户能从发布门禁页理解门禁会结合运行、评分、安全、审核和实验对比结果。
- 测试计划配置能解释门禁与安全审核链路。
- 不破坏已有 API 契约和项目隔离。

## 完成记录

- 人工审核页已增加运行来源、查看安全测试、发布门禁放行路径，空状态和任务详情均提供运行中心和门禁入口。
- 安全测试页已明确“已发布 Agent 版本 → 启动安全测试 → 查看运行证据 → 发布门禁判断”的流程，并将按钮改为“启动安全测试”。
- 发布门禁页已增加测试计划、运行中心、安全测试、人工审核和实验对比入口，评估按钮改为“评估运行结果”。
- 测试计划版本向导已在门禁配置步骤解释门禁会读取运行通过率、安全评分和待人工审核数量，并提供安全测试、人工审核和发布门禁入口。
- 后端链路已通过测试确认：运行结果会自动收集人工审核任务；门禁评估会读取运行、审核和安全证据并记录发布决策。
