# Codex 浏览器执行模型配置产品化

## 背景

Codex 浏览器探索链路已经跑通，但用户仍然容易把 OpenAI、Codex CLI 和 Chrome 浏览器实例混在一起理解。当前模型参数主要隐藏在用例 `input.model` / `input.model_provider` 中，测试计划配置只显式绑定浏览器实例，无法让用户清楚知道“Codex 用哪个模型执行浏览器探索”。

## 目标

让 Codex 浏览器探索的模型选择成为测试计划执行配置的一部分：

1. Chrome 浏览器实例继续只表达浏览器目录、登录态和 CDP 复用。
2. Codex 执行模型独立显示，默认使用 Codex CLI 当前配置，也允许选择平台模型配置。
3. 发布测试计划后，运行快照携带 Codex 模型配置，API Runner 自动传给 Codex Browser Activity。
4. 前端文案用“Codex 执行模型”，避免暗示浏览器必须依赖 OpenAI。

## 实施步骤

1. 扩展测试计划 `ExecutionConfig`，增加 `codex_model_config_id`、`codex_model`、`codex_model_provider`，并保持旧数据兼容。
2. 发布/运行快照将 Codex 模型配置写入 execution policy。
3. Worker 执行 `codex_explore` 时优先读取用例 input，其次读取测试计划 execution policy，最后回退到 Codex CLI 默认配置。
4. 前端测试计划版本配置弹窗增加“Codex 执行模型”选择区，展示“默认使用 Codex CLI 配置”和平台已有模型。
5. 补充前后端测试，验证配置传递、展示和默认文案。

## 验收

- 不选择 Codex 执行模型时，运行仍使用 Codex CLI 当前默认配置。
- 选择平台模型时，发布测试计划版本后运行快照包含 model/provider 信息。
- Worker 构造 `CodexBrowserTaskInput` 时能从 execution policy 读取 model/provider。
- 前端测试计划配置弹窗能选择 Codex 执行模型，文案清楚区分浏览器实例与模型。
- 相关单元测试、类型检查和构建通过。

## 风险

- 平台模型配置和 Codex CLI provider 名称不一定完全一致。本次只传递 `provider/model` 字符串，真正能否执行以 Codex CLI provider 配置为准，并在 UI 文案中提示需要兼容 Codex CLI。
- 不做数据库迁移，字段放入现有 JSON 配置，降低对历史数据影响。
